# Voice dictation MVP — scope decisions

**Date:** 2026-04-23  
**Goal:** Ship a SuperWhisper-like loop quickly on **macOS and Windows**: hotkey → dictate → clean text → inject into the focused app, with **portable** shared logic where possible.

## Interaction model: toggle (owner decision)

**Decision:** **Start** recording: user-configurable global shortcut. While recording, **Stop & process** and **Cancel / discard** are **two different user-configurable shortcuts** (owner choice): **Stop** runs STT → LLM → inject; **Cancel** drops captured audio with **no** API calls. Those two outcomes must **never** share the same binding without an explicit modifier—no ambiguous single-key “maybe stop maybe cancel.”

**Why toggle can be better:** Hands-free while speaking (no need to hold a chord), natural for longer rants, pairs well with a visible or audible “recording” indicator.

**Tradeoff vs hold-to-talk:** You need clear **state** (idle vs recording), easy **cancel**, and guardrails so the mic is not left open by mistake. Optional later: add hold-to-talk as a second mode.

**Recording feedback (owner decision):** **Tray / menu bar** icon/state for idle vs recording, plus **short start/stop sounds** when recording toggles on and off (respect system mute later if needed). Windows uses system tray semantics; same product behavior.

**Default shortcuts (owner decision):** Ship **suggested default** global bindings for **Start**, **Stop & process**, and **Cancel** so first launch works out of the box; user can **remap** all three in settings (avoid clashes with common IDE shortcuts where possible—exact keys TBD in implementation).

## Text injection: synthetic typing (owner decision)

**Decision:** Insert final text via **synthetic keyboard events** (e.g. per-character or small batches)—**macOS:** `CGEvent` (and text input sources as needed for Unicode); **Windows:** `SendInput` / `INPUT`—not paste, so the **clipboard is not the transport** for the dictated text. **Injection is inherently OS-specific;** only orchestration, adapters, and config stay portable.

**Why this can be better:** Avoids clobbering the user’s pasteboard; feels closer to “I typed this” in some apps.

**Tradeoffs:** Slower for long output; **Unicode and emoji** are more fiddly; some fields still reject synthetic input (same class of failures as paste).

**When typing fails (owner decision):** **Strict** — surface a clear **error** only; **do not** fall back to paste or touch the clipboard. The user retries in another field or changes app; keeps the “no clipboard” guarantee absolute for MVP.

**Note:** The current [`spike/mac_injection_spike.py`](../spike/mac_injection_spike.py) still uses **paste** to validate Accessibility + injection quickly; the product MVP should switch the injector to match this decision.

## Transcription and cleanup: user-chosen models (owner decision)

**Decision:** **Abstract providers away.** The app never assumes “one AI stack.” The user **independently** chooses **which model/provider** runs **transcription** (speech → raw text) and **which** runs **cleanup** (raw text → final text). Those two choices can differ (e.g. vendor A for audio, vendor B for wording)—same vendor for both is only one possible configuration.

**Persistence (example):** **Canonical config on disk** (JSON aligned with [`config/example-model-settings.json`](../config/example-model-settings.json)); loaded by the **Python** core. **Secrets:** OS store—**Keychain** (macOS), **Windows Credential Manager / DPAPI** (Windows)—via thin `platform_*` modules, not long-term plaintext in JSON. Optional export for power users.

## Settings UI: localhost web (owner direction)

**Direction:** Serve the settings experience as a **static web app** from **`http://127.0.0.1:<port>`** only while the user has opened settings (or explicitly enabled the server), not as a public site.

**Why this can be a good idea:** One **portable** UI stack (HTML/JS/CSS) for Mac and Windows; rich forms for two adapter panels, prompts, and docs links; fast iteration independent of native UI toolkits.

**What to watch:** **Security** — bind to **127.0.0.1** only (never `0.0.0.0`), use a **random port** or fixed port plus a **one-time session token** (or mutual auth) for any mutating API so other local processes cannot silently reconfigure the app. **Lifecycle** — start the server when opening settings, stop it when the window is closed (avoid leaving an open control plane). **Alternative with fewer moving parts:** bundle the same static files in an **embedded** WebView (WKWebView / WebView2) and use `http://127.0.0.1` *internally* or a custom `app://` scheme—same assets, optional no separate TCP listener.

The settings UI should still expose **two** clear sections: “Transcription” and “Cleanup,” each with provider + model (and credentials as needed). Exact fields may evolve; keep the **two-step split** stable.

**Implications for MVP build**

- **Transcription adapter:** One interface (`AudioData` or file URL → `String`); concrete types plug in behind user settings only. Implementations may include OpenAI-compatible speech APIs, Deepgram, Apple Speech (if acceptable), bundled/local Whisper via subprocess, etc.
- **Cleanup adapter:** One interface (`String` transcript → `String` cleaned); same idea—OpenAI/Anthropic/Groq-compatible HTTP, local Ollama base URL, etc. **Optional** user-editable system prompt per cleanup profile; still **not** hard-wired to one vendor in code.
- **Orchestrator:** Record → `transcriptionService` (user’s transcription model) → `cleanupService` (user’s cleanup model) → inject. No shared “global default LLM” unless the user explicitly configures both steps to use it.
- **Defaults:** Ship a **sensible default pair** so first launch works without editing JSON, but defaults are just preset adapter configs, not compiled-in vendor lock.

**Why abstract:** Privacy/latency/cost tradeoffs differ per person; transcription quality and “voice” of cleanup are different problems and should be **user-tunable** without shipping separate app flavors.

**Where transcription and cleanup run (owner):** There is **no** single product-wide answer like “always cloud” or “always local.” **Each step** runs wherever the user’s adapter configuration points—e.g. public cloud API, private base URL, or local HTTP/process. Local vs cloud is an **emergent property of user choice per adapter**, not a global toggle baked into the app.

## Portability (macOS + Windows)

**Principle:** Maximize **shared** code for **config schema**, **orchestration** (record → transcribe → cleanup → inject), **HTTP to adapters**, and **the settings web assets**. Isolate **minimum** per-OS code: global shortcuts, microphone capture, tray/menu bar, synthetic injection, permissions onboarding, and secure credential storage.

## Out of scope for MVP

- **iOS** (and other mobile), **menu bar / tray “polish”** beyond functional state, custom “modes” UI, context from screen, file transcription, streaming partial results, **App Store / notarization / Microsoft Store** packaging beyond what’s needed to run locally.

## Success criteria for MVP

End-to-end on **each supported OS**: user starts recording, speaks, presses **Stop & process**; within a few seconds the cleaned text appears at the caret in a typical app (browser, Slack, Notes / Windows equivalents). **Cancel** discards without inserting text or calling remote models.
